# iOS 多型号布局方案

这份指南用于把 Figma 单一画板翻译成当前仓库里可复用的 UIKit 页面布局方案，目标不是只在一个机型上对齐，而是保证小屏不裁切、主流机型接近设计、大屏不松散。

## 1. 适配目标

- 优先覆盖 iPhone 小屏、主流全面屏和 Pro Max 大屏
- 仍然保持页面与 Figma 的视觉关系
- 统一使用 `SnapKit`
- 复杂背景继续优先使用真实导出资源
- 默认视觉基准为 `375 x 812`

## 2. 默认高度分档

- `compact`
  - 约 `667pt` 及以下
  - 代表机型：`iPhone SE` 一类小屏
- `regular`
  - 约 `700-899pt`
  - 代表机型：主流全面屏
- `expanded`
  - 约 `900pt` 及以上
  - 代表机型：`Pro Max`

当前仓库已把这套分档抽到：
- `XiandaoDemo/Common/Utils/Constants.swift`

当前默认基准也在这里：
- `AdaptiveLayout.referenceScreenHeight = 812`

## 3. 布局原则

### 纵向布局

- 不把整页长期写成固定 `y` 坐标
- 把 Figma 的绝对坐标翻译成“模块相对约束”
- 只保留少量档位化常量，比如：
  - 顶部间距
  - 模块之间的竖向间距
  - 背景资源尺寸
  - 关键按钮高度

### 基准使用方式

- 首版视觉以 `375 x 812` 为参考
- `regular` 档应最接近 Figma 设计关系
- `compact` 档主要负责收紧留白和避免裁切
- `expanded` 档主要负责防止页面过空

### 横向布局

- 按左右内边距控制内容宽度
- 尽量避免写死整块固定宽度
- 如果内容在大屏上过宽，优先收紧边距或限制最大宽度

### 长页面

- 表单页、登录页、注册页、设置页默认使用 `UIScrollView`
- 小屏下优先保证内容可完整滚动，不强行维持单屏完全展示

### 背景和插画

- 复杂矢量背景优先导出真实资源
- 单独导出的资源要先判断：
  - 是否按节点 frame 导出
  - 是否按图形边界裁剪
  - 是否带透明留白
- 如果是裁剪资源，位置优先按资源实际边界落位，不直接套 JSON 原始 frame

## 4. 当前仓库推荐实现方式

### 启动页类页面

- 可以用 `UIScrollView + contentView`
- 用弹性 spacer 或档位化间距控制纵向留白
- 主视觉居中，但在小屏上允许压缩空白区

### 表单类页面

- 背景资源放在根视图层
- 表单内容放进 `UIScrollView`
- 标题、上传区、表单区、主按钮、次按钮使用相对纵向链路
- 小屏通过滚动解决裁切问题，大屏通过档位化间距避免过空

## 5. 交付前必须检查

- 小屏：内容是否完整可见或可滚动到完整内容
- 主流机型：是否接近 Figma 设计关系
- 大屏：是否出现明显漂浮、空洞、纵向比例失衡
- 是否仍残留大量固定 `y` 坐标
- 是否把 `SnapKit` 和原生 Auto Layout 混用了
