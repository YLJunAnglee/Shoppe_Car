# 开发完成检查清单

这份清单用于每个页面或阶段性功能完成后的复查，目标是让交付结果可运行、可说明、可继续接下一阶段。

## 1. 页面功能检查

- 页面是否能正常进入
- 页面文案是否与设计稿一致
- 页面主要视觉元素是否已经接入
- 主要按钮和点击区域是否可点击
- 页面动画是否正常
- 页面空白态、占位态是否清楚

## 2. 页面流转检查

- 页面入口是否正确
- 点击主操作后的跳转是否正确
- 返回路径是否正确
- 登录态和未登录态的分流是否正确
- 根路由切换后页面是否符合预期

## 3. 资源检查

- Figma 导出的图片资源是否已经放入 `Assets.xcassets`
- 资源命名是否按功能命名，而不是按图层原名命名
- 页面是否已替换掉临时占位图
- 图标、插画、按钮图是否显示正常
- `Assets.xcassets` 是否存在异常资源
- 是否排查过伪装成 PNG 的 SVG 资源
- 对单独导出的资源，是否核对过像素尺寸和透明留白
- 资源本身如果已经包含边框、虚线、轮廓，代码里是否避免重复绘制

## 4. UI 还原检查

- 页面整体结构是否与 Figma 一致
- 标题、描述、按钮层级是否正确
- 关键间距、圆角、颜色是否大致对齐
- 中心插画或核心视觉是否已接近真实效果
- 不同机型下是否有明显错位或裁切
- 是否按 `compact / regular / expanded` 三档高度验证过布局
- 小屏设备下是否能完整看到内容，或至少可滚动到全部内容
- 大屏设备下是否避免出现明显过空、元素漂浮或纵向失衡
- 是否明确本页是按 `375 x 812` 默认基准，还是按新的指定机型基准实现
- 页面布局约束是否统一使用 `SnapKit`
- 是否仍残留手写 `NSLayoutConstraint` 或混用约束写法
- 复杂背景是否优先使用真实导出资源，而不是代码近似图形
- 如果页面参考的是整页绝对坐标，是否已经翻译成可适配的相对纵向约束
- 是否仅把高度档位常量作为参考，而不是把所有控件继续写成固定 `y`

## 5. 工程接入检查

- 新页面是否已正确加入工程
- 是否接入了现有导航流
- 是否接入了现有登录态或业务状态
- 入口是否仍依赖旧 storyboard，还是已经切换到代码入口
- 生成代码是否错误纳入 target

## 6. 编译检查

- Xcode GUI 是否能成功编译
- 是否存在 Swift 6 并发警告或错误
- 是否存在资源编译错误
- 是否存在因自动生成代码导致的无关编译错误
- 是否区分清楚代码问题和本机环境问题

## 7. 文档检查

- 是否新增本阶段交付文档到 `docs/stages/`
- 是否更新 `docs/pages/page-map.md`
- 是否写清本阶段新增页面、修改页面和占位页面
- 是否记录本阶段处理过的关键问题
- 是否写出下一阶段建议

## 8. 建议记录结果的方式

每次阶段文档建议至少给出以下结论：

- 本阶段完成状态：已完成 / 部分完成 / 占位完成
- 当前可演示路径：从哪个页面进入，到哪个页面结束
- 当前遗留问题：哪些页面仍未接真实设计稿
- 下一步接入建议：下一个最合理的页面或流程

## 9. 当前仓库默认适配策略

- 设备高度分 3 档：
  - `compact`：约 `667pt` 及以下
  - `regular`：约 `700-899pt`
  - `expanded`：约 `900pt` 及以上
- 视觉基准默认是 `375 x 812`
- 表单页、长页面默认使用 `UIScrollView`
- 纵向布局优先使用模块相对约束，不长期保留整页绝对坐标
- 复杂背景优先用真实资源，资源位置按导出方式和实际边界决定
